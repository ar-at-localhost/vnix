** DONE Persist current workspace state
   - Let the user choose 2 config files
     - *Primary*
       A primary specs file contains user's defined (maybe Nix generated) specs
       for initial reads (definitions). It can be either JSON, or Nix (though
       the generated primary source is always Nix).

       It has 2 parts, the readonly path and the write path.
       Readonly path is ~$HOME/.vnix/vnix.json~. Vnix doesn't overwrite it.
       Write path is specified by config option ~specs_file_primary~. Vnix will write it and forget.
       
       Vnix auto-reads Primary source if secondary source isn't available.
       Keys: ~<leader>P~ Persist primary source
    - *Secondary*
       A secondary spec is reflection of runtime specs. It is beneficial for restoring,
       modified state of Vnix upon next start. It can be overwritten by manually saving,
       or future auto-save option.
       Keys: ~<leader>p~ Persist secondary source

    *TLD;R*
    Primary source is a readonly backup. Define your workspaces in primary source file.
    Feel free to create new panes / tabs or workspaces on the fly. Keep your changes in secondary source.
    When you wish to make them primary source - manually save.

    *Why*
    It let you keep declarative manners of Nix. Define workspaces in your config.
    Don't run to declarations, just for saving another pane, keep saving in secondary
    save. When you're going to rebuild system (or home), save it to primary source if you
    generate new config based on it.

*** Original Ideas
   If user makes changes to the running panes, i.e. deleting a pane, tab or workspace,
   it is currently only affecting the session and upon next session, it is restored from
   workspaces's file. That's perfect to make things persistence, however has an undesired
   side-effect or losing decisions.

   There should be some way to persist back the current state.
   One way would be to patch the workspace's file (a ~JSON~) but
   would defeat the declarations of Nix's workspace file specs.

   Maybe we can choose between the two:
   A. Let the user forcefully overwrite the workspace's file.
   B. Update the user specified Nix module to persist declarations back.

   For example, I've many workspaces declared through Nix. While at run-time,
   I find a tab unnecessary and just delete it. Next time I run Vnix, it is back,
   since it did affect the state, but not workspace's file. How to persist such
   an action?
   - Provide a shortcut to persist to currently set workspace file.
   - Allow user to choose the Nix module location, where it keeps its declaration,
      upon certain action, generate a Nix expression (attrSet) translated from
     current state to Nix specs. User can re-build it later to persist the decisions.
     This will also let the user actually review its decision before committing.
     This is quite awkward solution, but it aligns with VNix philosophy of declarative
     workspaces and separation of concern between run-time state and declarations.
